Signed - 최상위 비트(MSB)를 부호를 표시하는 데 사용
Unsigned - 모든 비트를 전부 숫자의 크기를 표현하는 데 사용

-> 음수를 표현할 수 있느냐의 차이이며 대신 표현할 수 있는 양수의 최댓값이 2배로 늘어나는 것이기에 메모리 용량은 동일

// 리눅스 & macOS (LP64 모델): 64bit 모델로 넘어가면서 포인터도 8바이트니까, long 사이즈도 8byte로 변경
// 윈도우 (LLP64 모델): 32비트 시절에 long을 4바이트로 정의하여 사용했기다 long은 그냥 4바이트로 유지

// sizeof 연산자의 결과값은 size_t라는 타입으로 나오기 떄문에, %d 대신 %zu라는포맷을 사용하는 것이 좋음

Challenge-3

결과

Signed int (INT_MAX + 1)
-> 2147483647 + 1 ➔ -2147483648 (양수 최댓값이 음수 최솟값으로 감소)

Unsigned int (UINT_MAX + 1)
-> 4294967295 + 1 ➔ 0 (양수 최댓값이 0으로 초기화)

원인

4byte의 메모리 크기를 초과하는 값을 저장하려 할 때 발생하는 정수 오버플로우(Integer Overflow) 데이터가 할당된 공간을 초과하면서 최상위 비트(MSB)의 부호가 완전히 뒤집히거나, 초과한 비트가 밖으로 버려지면서 0으로 초기화

Challenge-4

실험 및 결과

int 변수를 문자열 출력 포맷인 %s로 출력 시도 하였고 프로그램이 비정상 종료되며 Segmentation fault (core dumped) 에러 발생

원인

printf 함수에서 %s는 값을 메모리 주소로 취급하여, 그 주소로 부터 문자열의 끝을 의미하는 NULL 바이트를 찾을 때 까지 문자열을 읽음

하지만 일반 정수를 %s에 넘기게 되면, 컴퓨터는 이를 메모리 주소로 착각하고 접근을 시도하고 해당 주소는 프로그램이 접근할 권한이 없는 구역이므로, 운영체제가 메모리 보호를 위해 프로그램을 강제 종료시킴

Format String Bug

만약 printf(user_input); 형태로 코드를 작성하게 될 시 문제가 발생할 수 있음

1. 메모리 유출 (Memory Leak): 입력창에 %x %x %x %x를 입력하면, 프로그램은 포맷 스트링에 매칭되는 변수가 없음에도 스택(Stack) 메모리 상에 있는 값들을 출력 -> 의도하지 않은 데이터가 유출될 가능성이 있음

2. 임의 주소 쓰기 (Arbitrary Memory Write): %n은 지금까지 출력한 글자의 개수를 메모리에 저장하는 명령어인데, 이를 악용하여 메모리의 특정 주소에 자신이 원하는 값을 강제로 덮어써서, 프로그램의 실행 흐름을 조작하고 서버의 쉘을 탈취할 수 있음
